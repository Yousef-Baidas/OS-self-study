## What Is an Operating System?

<Lead from={6} to={11}>The one program that is always running — it manages the hardware and stands between you and the machine.</Lead>

An **operating system** is a program that manages a computer's hardware, provides a base for
application programs, and acts as an **intermediary** between the user and the hardware <SlideRef n={6} />. A computer
system has four layered components <SlideRef n={3} />:

| Component | What it contributes |
| --- | --- |
| Hardware | Basic computing resources — CPU, memory, I/O devices |
| Operating system | Controls and coordinates the hardware among apps and users |
| Application programs | Define how the resources solve users' problems — browsers, compilers, games |
| Users | People, machines, or other programs |

<Figure
  src="figures/ch01/system-structure.svg"
  alt="System structure diagram: users and their application programs sit on top of the operating system, which sits directly on top of the computer hardware"
  caption="Figure 1.1 — Users and programs are layered above the operating system, which sits directly above the hardware."
/>

**Three goals** shape everything the OS does <SlideRef n={7} />:

- Run user programs and make solving problems easier.

- Make the computer system **convenient** to use.

- Use the hardware **efficiently**.

Whose interests come first depends on who's asking <SlideRef from={8} to={9} />:

| Question | User view | System view |
| --- | --- | --- |
| What matters most? | Convenience, ease of use, performance | Efficient, fair use of resources |
| Cares about utilization? | No | Yes |
| So the OS looks like… | A helpful intermediary | A **resource allocator** |

<Callout type="note">
  The OS wears two hats at once <SlideRef n={10} />. As a **resource allocator** it decides who gets the CPU,
  memory, storage, and I/O devices — and for how long. As a **control program** it manages how
  user programs run, so one misbehaving program can't take down the whole machine.
</Callout>

That gives the working definition we'll use throughout the course: the OS is **the one program
always running** on the computer — the **kernel** <SlideRef n={11} />. Everything else is either a **system
program** (ships with the OS but isn't in the kernel) or an **application program** (everything
not tied to the OS's own operation). You've already used several: Windows, macOS, Linux, Unix,
and Ubuntu — some commercial, some open-source <SlideRef n={5} />.

## Computer System Organization

<Lead n={12}>One or more CPUs and a set of device controllers share a common bus that reaches shared memory.</Lead>

- **CPUs** and **device controllers** all connect through a **common bus** to **shared memory**.

- Each **device controller** manages one type of device (a disk, a USB port, a graphics adapter) and has a local buffer.

- CPUs and controllers run **concurrently**, competing for memory cycles across that shared bus.

## Computer Startup

<Lead n={13}>At power-on, a firmware bootstrap program wakes the hardware and loads the kernel.</Lead>

The **bootstrap program** is the first code to run when a machine is powered on or rebooted <SlideRef n={13} />. Its job:

- **Initialize** every part of the system — CPU registers, device controllers, memory contents.

- **Find and load** the operating-system **kernel**, then start executing it.

<dl>
  <dt>Firmware</dt>

  <dd>Read-only memory (ROM or EPROM) built into the hardware. The bootstrap lives here because it must be reliable <strong>before</strong> any disk has been read.</dd>
</dl>

This is exactly what you watch at startup: a manufacturer boot screen (firmware), then the OS's
own loading screen as the kernel comes up into memory.

## Storage-Device Hierarchy

<Lead from={16} to={20}>Storage is layered by speed, cost, and volatility — and where a piece of data sits tells you how the system uses it.</Lead>

Storage is organized into a **hierarchy** ordered by speed, cost, and volatility <SlideRef n={16} />. The single most
exam-worthy line in that hierarchy is the **volatile / nonvolatile boundary**:

- **Volatile** — lost the instant power is removed: **registers** → **cache** → **main memory (RAM)**.

- **Nonvolatile** — survives a power cycle: **NVM / SSD** → **hard disk** → **optical disk** → **magnetic tape** (slowest, backups only).

Two tiers you must be precise about <SlideRef n={18} />:

- **Main memory (RAM)** — the *only* large storage the CPU can access **directly**; random-access; **volatile**.

- **Secondary storage** — a **nonvolatile** extension of main memory, for data that must outlive a power cycle.

Access is neighbor-to-neighbor: registers reach cache, cache reaches main memory, main memory
reaches the disk — rather than every level reaching all the way down on its own.

<Callout type="note">
  A **hard-disk drive** stores data magnetically on rigid rotating platters, divided into tracks
  and sectors and run by a disk controller <SlideRef n={19} />. An **SSD** (solid-state drive) is nonvolatile like a
  hard disk but built on different technology, and is faster for most workloads.
</Callout>

Sizes use standard units <SlideRef n={20} />: a **bit** is one 0 or 1, a **byte** is 8 bits, and a **word** is
whatever width the architecture works with natively. Beyond a byte, capacity scales in powers of
1024 — KB, MB, GB, TB, PB.

## Multiprocessing and Clustered Systems

<Lead from={21} to={25}>Add processors — even whole machines — for more throughput, better economy, and higher reliability.</Lead>

**Parallel systems** (also called multiprocessor or tightly coupled systems) use more than one CPU
in close communication, sharing the bus and often the clock, memory, and devices <SlideRef n={21} />. Three advantages:

- **Increased throughput** — more work per unit time.

- **Economy of scale** — sharing peripherals, storage, and power costs less per processor.

- **Increased reliability** — one processor fails, the system **degrades gracefully** instead of stopping.

There are two ways to organize the processors <SlideRef n={22} />:

| Aspect | Asymmetric multiprocessing | Symmetric multiprocessing (SMP) |
| --- | --- | --- |
| Processor roles | Each assigned a specific task | Every processor is a peer |
| OS copies | One master controls the rest | Each processor runs its own copy |
| Memory | — | All processors share the same memory |

A **dual-core design** takes SMP onto a single chip — multiple **cores** sharing memory, rather
than multiple separate chips <SlideRef n={23} />.

**Clustered systems** go a step further: multiple *complete* computers work together, usually
sharing storage over a **storage-area network (SAN)**, to deliver **high-availability** service
that survives a node failure <SlideRef from={24} to={25} />. Clustering comes in an **asymmetric** form (one node in
**hot-standby**, ready to take over) and a **symmetric** form (all nodes active, monitoring each
other). Clusters also power **high-performance computing (HPC)**, coordinated by a **distributed
lock manager (DLM)** that arbitrates shared-resource access across nodes.

## Multiprogramming and Multitasking

<Lead from={26} to={27}>Never let the CPU sit idle — batch multiprogramming first, then interactive multitasking.</Lead>

A single CPU runs one instruction at a time, so keeping it busy is what makes a system efficient.
These two techniques are a classic exam distinction:

| Aspect | Multiprogramming (batch) | Multitasking (timesharing) |
| --- | --- | --- |
| Main goal | Keep the CPU busy — efficiency | Interactive, live response |
| Switches when… | The running job has to wait (e.g. for I/O) | So frequently users interact with each job as it runs |
| Scheduling decision | **Job scheduling** — which job in memory runs | **CPU scheduling** — which ready process runs next |
| Response time | Not a concern | Feels immediate (`<1s`) |

Supporting vocabulary that rides along with multitasking <SlideRef n={27} />:

- **Process** — a user's program in execution.

- **Swapping** — move a process temporarily out to disk to free memory.

- **Virtual memory** — run processes that aren't even fully loaded into memory.

## Interrupts

<Lead from={29} to={32}>The OS is interrupt-driven — hardware and software signals divert the CPU to a handler, then it resumes.</Lead>

Interrupts come from two directions <SlideRef n={29} />:

- **Hardware interrupt** — raised by a device (e.g. an I/O operation finishing), delivered as a signal to the CPU over the bus.

- **Software interrupt** — also called a **trap** or **exception**, triggered by the running program itself: an error (like division by zero) or a deliberate request for an OS service (a **system call**).

**How an interrupt is serviced** <SlideRef from={30} to={31} />:

- Control transfers through the **interrupt vector** — a table holding the address of every interrupt service routine.

- Before handing control over, the hardware **saves the address of the interrupted instruction**, so execution resumes exactly where it left off.

<Callout type="note">
  Because it reacts to events this way, an operating system is fundamentally **interrupt driven**.
  A **trap** is just a software-generated interrupt — caused by an error or by a request for a
  service — and it's the mechanism the OS uses to catch problems like an infinite loop or one
  process interfering with another.
</Callout>

## Operating-System Operations and Dual Mode

<Lead from={33} to={36}>Hardware dual-mode operation lets the OS protect itself and every other program from a misbehaving process.</Lead>

The CPU runs in one of two modes, tracked by a single hardware **mode bit** <SlideRef n={34} />:

| Aspect | User mode | Kernel mode |
| --- | --- | --- |
| Mode bit | `1` | `0` |
| What runs | Application code | The operating system itself |
| Privileged instructions | ✗ Rejected (traps) | ✓ Allowed |
| Also called | — | Supervisor / system / privileged mode |

- **Privileged instructions** execute only in kernel mode; attempting one in user mode traps <SlideRef n={35} />.

- A **system call** is how user code deliberately flips the mode bit to kernel, so the OS can do a privileged operation on its behalf — then the bit flips back on return.

<Figure
  src="figures/ch01/mode-transition.svg"
  alt="Diagram of the transition from user mode to kernel mode: a user process executes, calls a system call which traps and sets the mode bit to kernel, the kernel executes the system call, then returns and resets the mode bit to user"
  caption="Figure 1.4 — A system call traps into the kernel, then returns and resets the mode bit to user."
/>

Above the kernel sits the **shell** — the user interface, whether GUI or command line <SlideRef n={36} />. The kernel
itself is organized into **seven** core responsibility areas: process management, memory
management, storage management, the file system, CPU scheduling, I/O management, and networking &
security.

## Kernel Data Structures

<Lead from={37} to={39}>The kernel tracks its own state with the same standard structures you'd meet in any data-structures course.</Lead>

- **Linked lists** <SlideRef n={37} /> — elements chained by pointers: **singly** (each links to the next), **doubly** (links to both neighbors), **circular** (last links back to first).

- **Binary search tree** <SlideRef n={38} /> — every node's left child ≤ the node ≤ its right child. Search is O(n) worst case, but a **balanced** BST keeps it to **O(lg n)**.

- **Hash map** <SlideRef n={39} /> — a hash function maps a key to a slot for near-constant-time lookup.

- **Bitmap** <SlideRef n={39} /> — a string of *n* bits, each recording the on/off status of one item (e.g. which disk blocks are free).

The Linux kernel ships real, minimal versions of exactly these, in the headers the slide points to:

```c title="linux-kernel-data-structures.c" {2,10,18}
// <linux/list.h> — the doubly linked list every kernel subsystem embeds
struct list_head {
  struct list_head *next, *prev;
};

// <linux/rbtree.h> — the balanced tree backing kernel maps/schedulers
struct rb_node {
  unsigned long __rb_parent_color;
  struct rb_node *rb_right;
  struct rb_node *rb_left;
};

// <linux/kfifo.h> — the generic ring-buffer FIFO
struct __kfifo {
  unsigned int in, out;   // write/read positions
  unsigned int mask;      // size - 1 (power-of-2 sizing)
  unsigned int esize;     // element size
  void *data;             // buffer pointer
};
```

<Callout type="note">
  Notice `list_head` and `rb_node` hold no payload of their own — they're embedded directly
  inside the struct that owns the data. This is a Linux idiom called an **intrusive data
  structure**: the list/tree machinery lives next to the data it organizes, so the kernel avoids a
  second allocation just to link something into a list.
</Callout>

<Callout type="key">
  **Key takeaways:** the OS is the one program always running — the **kernel** — acting at once as
  **resource allocator** and **control program**. Hardware organizes around a shared **bus**
  linking CPUs, controllers, and memory, and boots via a firmware **bootstrap program**. Storage
  is a **hierarchy** split by the **volatile / nonvolatile** line. Systems scale out through
  **multiprocessing** and **clustering**, and scale CPU use through **multiprogramming** and
  **multitasking**. The CPU stays responsive through **interrupts** (via the **interrupt vector**)
  and protects itself with **dual-mode** operation enforced by a hardware **mode bit**.
</Callout>
